# Fertilizer / Supply Application — Design

**Status:** approved for planning (2026-04-21)
**Target:** Verdant backend (Quarkus + Kotlin), Verdant web admin (React), Verdant Android (Compose)
**Driver:** record fertilizer applications against beds or specific plants, decrement supply inventory atomically with the record, surface applications on plant timelines and bed history, and let species workflow plans include fertilizer steps with suggested supply + quantity.

---

## 1. Scope

Fertilizer is the first consumer of a **generic "supply application"** subsystem. The schema, endpoints, and activity screen are product-agnostic — any `SupplyType` can flow through them. The only fertilizer-specific piece in MVP is the entry-point label ("Fertilize") and a default filter to `category = FERTILIZER` in the supply picker. When pesticide / compost / foliar feed / mulch need the same treatment later, we add sibling entry points and optional category filters — no schema churn.

**Not in scope for MVP:**

- Dedicated pesticide / mulch / compost entry points. The generic infrastructure is there; the specific entry points aren't built yet.
- `DELETE` or `PUT` on `supply_application`. Recorded-in-error cases are handled by manual inventory adjustment + deleting the plant_event rows on the plant timeline.
- Recurring / scheduled fertilization independent of workflow. The workflow step + `daysAfterPrevious` already handles cadence.
- Rule-based supply suggestions ("any balanced NPK"). Workflow steps point at specific `SupplyType` rows.
- Per-m² dose calculations off bed length × width. The `raised_bed` + dimensions columns already exist; autoscaling a recommended quantity lives later.
- Reporting dashboards ("fertilizer spend by season"). `Analytics` can grow this later; out of scope now.

**Existing infrastructure kept as-is:**

- The existing Android `SupplyUsageBottomSheet` — a fast post-activity decrement flow — stays. It's inventory housekeeping (soil, trays, labels) without audit, used when traceability isn't wanted. Users learn which path to take for which situation. No retrofitting of that flow to write `supply_application` rows.

## 2. Data model

### New migration: `V20__supply_application.sql`

**`supply_application`** — canonical record of one application event.

| Column | Type | Notes |
|---|---|---|
| `id` | `BIGINT GENERATED BY DEFAULT AS IDENTITY PK` | |
| `org_id` | `BIGINT NOT NULL REFERENCES organization(id) ON DELETE CASCADE` | |
| `bed_id` | `BIGINT NOT NULL REFERENCES bed(id) ON DELETE CASCADE` | Always required — plant-level apps still target a specific bed |
| `supply_inventory_id` | `BIGINT NOT NULL REFERENCES supply_inventory(id)` | Specific lot |
| `supply_type_id` | `BIGINT NOT NULL REFERENCES supply_type(id)` | Denormalised from the inventory row for easy filtering |
| `quantity` | `NUMERIC(12, 2) NOT NULL CHECK (quantity > 0)` | In the supply's native unit |
| `target_scope` | `VARCHAR(8) NOT NULL CHECK (target_scope IN ('BED','PLANTS'))` | Bed-level vs plant-level |
| `applied_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | |
| `applied_by` | `BIGINT NOT NULL REFERENCES app_user(id)` | |
| `workflow_step_id` | `BIGINT REFERENCES species_workflow_step(id) ON DELETE SET NULL` | Nullable — set when triggered from workflow completion |
| `notes` | `TEXT` | |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | |

Indexes:
- `(bed_id, applied_at DESC)` — bed history
- `(supply_inventory_id)` — supply audit
- `(org_id, applied_at DESC)` — org-wide export / list

### Extend existing `plant_event`

- Add `APPLIED_SUPPLY` to the `PlantEventType` enum.
- Add nullable column `supply_application_id BIGINT REFERENCES supply_application(id) ON DELETE SET NULL`.
- Plant-level applications create one `plant_event` per targeted plant with `event_type = 'APPLIED_SUPPLY'` and `supply_application_id` set. Bed-level applications create none.
- On plant delete: `plant_event` rows cascade (existing behaviour). `supply_application` row survives — the bed-level record is intact.

### Extend existing workflow step tables

Both `workflow_template_step` and `species_workflow_step`:

- Add nullable `suggested_supply_type_id BIGINT REFERENCES supply_type(id) ON DELETE SET NULL`.
- Add nullable `suggested_quantity NUMERIC(12, 2)`.
- Populated only when the step's `event_type = 'APPLIED_SUPPLY'`. Other step types leave them null. No CHECK constraint ties them to the event type — the app layer enforces it at create / update time.

### Authorization

Every query / mutation on `supply_application` verifies `supply_application.org_id = caller's org`. Cross-org access returns `404` (not 403 — leak less).

## 3. Flow semantics

### `POST /api/supply-applications`

Request body:

```json
{
  "bedId": 42,
  "supplyInventoryId": 7,
  "quantity": 60.0,
  "targetScope": "PLANTS",
  "plantIds": [101, 102, 103, 104, 105],
  "workflowStepId": 88,
  "notes": "Overcast, 15°C"
}
```

Single DB transaction, in order:

1. **Authorization** — bed, supply inventory, every plant in `plantIds`, and (if set) the workflow step all belong to the caller's org. Any mismatch → 404.
2. **Validation:**
   - `quantity > 0` (400).
   - `quantity <= supply_inventory.quantity` (400: "Insufficient quantity," matching existing `decrementInventory`).
   - `targetScope = 'BED'` requires `plantIds` empty/null and `workflowStepId` null (workflow completion is inherently per-plant).
   - `targetScope = 'PLANTS'` requires non-empty `plantIds`; every id must currently reference `bed_id = bedId`; status ≠ `REMOVED`.
   - If `workflowStepId` is set, the step must match a species that at least one of the target plants is.
3. **Decrement** `supply_inventory.quantity` by `quantity` (via existing `SupplyInventoryRepository.decrementQuantity` in the same transaction).
4. **Insert** `supply_application` row. `supply_type_id` is copied from the inventory row.
5. **Fan out** — if `targetScope = 'PLANTS'`, insert one `plant_event` per `plantId` with `event_type = 'APPLIED_SUPPLY'`, `supply_application_id = <new id>`, `event_date = applied_at::date`.
6. **Workflow progress** — if `workflowStepId` is set, insert one `plant_workflow_progress(plantId, stepId)` row per plantId.
7. **Commit.** Response: `SupplyApplicationResponse` with the new id, echoed fields, resolved supply name, unit, and lot remaining after decrement.

Any failure rolls the transaction back — no partial decrement, no orphan events.

### Read endpoints

- `GET /api/supply-applications/bed/{bedId}` — list recent applications for one bed (paginated, default 20, max 100). Used by BedDetail.
- `GET /api/supply-applications/garden/{gardenId}` — list across all beds in the garden, paginated.
- `GET /api/supply-applications/{id}` — single application, resolving plant list when `targetScope = 'PLANTS'`.

### No `DELETE` or `PUT` in MVP

Error corrections go through manual inventory adjustment + deleting the `plant_event` rows via the plant timeline. A proper reversal endpoint would need to re-credit inventory atomically and handle deleted-plant cases; defer until a real user asks.

## 4. Workflow integration

Template step editor and species step editor both gain two conditional fields, visible only when `event_type = 'APPLIED_SUPPLY'`:

- **Supply type picker** — dropdown over the org's `supply_type` rows. A category filter (`FERTILIZER`) is applied by default but can be cleared.
- **Suggested quantity** — numeric input; unit label from the picked supply type.

`SpeciesWorkflowStep` overrides the template's suggestion per the existing pattern (same as the other step fields).

Step completion flow:

- Clicking "Complete" on an `APPLIED_SUPPLY` step does **not** auto-record the application. It routes to `activity/apply-supply?bedId=X&plantIds=...&stepId=Y&supplyTypeId=Z&quantity=Q`.
- User picks the actual inventory lot, optionally adjusts quantity, confirms.
- On submit: `supply_application`, per-plant `plant_event`s, inventory decrement, and `plant_workflow_progress` rows all land in one transaction.
- On cancel: nothing persists. Step stays incomplete.

## 5. UI surfaces

### Shared activity screen

Route: `activity/apply-supply` (web) / `ApplySupplyScreen` (Android).

Query params / nav args: `bedId` (required), `plantIds` (optional CSV / LongArray), `supplyTypeId` / `quantity` (optional suggestions from workflow), `stepId` (optional, passed through to POST).

Layout:

1. **Target toggle** — radio `Whole bed / Specific plants`. Defaults to "Specific plants" if `plantIds` in URL.
2. **Plant selection** — visible only in PLANTS mode. Checklist of plants currently in the bed (status ≠ REMOVED). Pre-selected from `plantIds`.
3. **Supply picker** — searchable dropdown over `supply_inventory` rows with `quantity > 0`, grouped by `supply_type.name`. Each row shows product name, lot's remaining quantity + unit, cost if set. Filtered to `FERTILIZER` by default; "show all supplies" toggle lifts the filter.
4. **Quantity input** — number with the supply's unit label adjacent. Soft validation: red hint when `quantity > lot.remaining`. Pre-filled if query param set.
5. **Notes** (optional textarea).
6. **Submit** — button label reflects the entry point ("Fertilize" when opened from that button, "Apply" when opened from the generic path). Success toast + navigate back.

### Entry points

**Web:**

- `BedDetail` — "Fertilize" button near "Add plant" → `activity/apply-supply?bedId=X`.
- `BedDetail` — "Applications" section below plants, listing the 10 most recent `supply_application` rows for this bed.
- `PlantDetail` — existing event timeline gains an `APPLIED_SUPPLY` renderer ("Applied 20g of Rosafluid"). Click opens a simple modal with supply, quantity, other plants hit, notes.
- `WorkflowTemplateEdit` / `SpeciesWorkflowEdit` — step editor exposes the suggestion fields when event type is `APPLIED_SUPPLY`.
- Workflow completion on an `APPLIED_SUPPLY` step routes to the activity.

**Android:**

- `BedDetailScreen` — "Fertilize" button, "Applications" section.
- `PlantDetailScreen` — timeline case for `APPLIED_SUPPLY`.
- Plant picker activities (the one used for harvest / pot-up) gain a "Fertilize" variant.
- Workflow completion — same deep link as web.

Swedish register: horticultural, matching prior translations (*Gödsla* for the verb, *Applicering* for the noun).

## 6. Testing strategy

### Backend

- **`SupplyApplicationServiceTest`** (unit, Mockito-kotlin):
  - rejects `quantity > lot.quantity` (400, no decrement)
  - rejects cross-org bed / supply / plant / step (404)
  - rejects `PLANTS` with empty `plantIds` (400)
  - rejects `BED` with non-empty `plantIds` (400)
  - rejects plant not in bed (400)
  - rejects plant with status `REMOVED` (400)
  - rejects `stepId` with `BED` scope (400)
  - happy-path `PLANTS`: inserts one `supply_application`, decrements inventory, inserts N `plant_event`, inserts N `plant_workflow_progress` when `stepId` set, all in one transaction
  - happy-path `BED`: inserts one `supply_application`, decrements inventory, zero `plant_event`, rejects `stepId`
- **`SupplyApplicationRepositoryTest`** (`@QuarkusTest`, real Postgres): transactional rollback — seed insufficient inventory, call service, assert no `supply_application` row persists.
- **`SupplyApplicationResourceTest`** (service-level unit; no `quarkus-test-security`): authorization 404s on cross-org, shapes of POST / GET responses.

### Web

- Component test on the apply-supply form: target toggle swaps plant selection on/off; supply picker filters to FERTILIZER by default and shows lot remaining; quantity hint appears on exceed; submit calls `api.supplyApplications.create` with the correct body. `msw` for the API.

### Android

- ViewModel unit tests for `ApplySupplyViewModel`: loads supplies, computes bed vs plants body, handles insufficient-quantity error.

### Manual verification

- Fertilize a bed from `BedDetail`. Confirm the "Applications" section updates.
- Fertilize three specific plants from the plant picker. Confirm each plant's timeline shows the event.
- Complete a workflow step with an `APPLIED_SUPPLY` suggestion. Confirm the activity opens pre-filled.
- Try to apply more than a lot holds. Confirm 400 and no inventory change.

## 7. Phasing

Five tasks, single plan:

1. **Schema + entity + DTO + repository.** V20 migration. `SupplyApplication` entity. Extend `PlantEvent` (enum value + nullable FK). Extend workflow step entities (two nullable columns). Repository methods: `insert`, `findByBed`, `findByGarden`, `findById`. Unit tests for the repository against `@QuarkusTest`.
2. **Service + resource.** `SupplyApplicationService` with the full transactional flow and authorization. `SupplyApplicationResource` exposing the endpoints. Full test suite. `SupplyInventoryRepository.decrementQuantity` is already present.
3. **Web — activity screen + entry points.** `ApplySupply.tsx`. BedDetail "Fertilize" button + "Applications" section. PlantDetail timeline case. WorkflowTemplateEdit / SpeciesWorkflowEdit conditional fields. Workflow step completion routing.
4. **Android — activity screen + entry points.** `ApplySupplyScreen.kt`. BedDetailScreen button + Applications section. PlantDetailScreen timeline case. Plant picker variant. Workflow completion routing.
5. **Data export + milestone.** Add `supplyApplications` to the org export DTO in `DataExportService`. End-to-end smoke. Milestone commit.

Rough sizing: ~4.5 solo-dev days.
