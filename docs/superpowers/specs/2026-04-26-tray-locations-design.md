# Tray Locations Design

## Goal

Let users tag where their physical trays sit (e.g. "Greenhouse", "Cold frame", "Kitchen window"), require a location pick when they sow into a tray and 2+ locations exist, and let them bulk-act (water, fertilize, note, move) on every tray plant in a location.

## Non-goals

- Per-tray entities (a tray is still implicit — the bag of `bed_id IS NULL` plants).
- Capacity tracking, environmental sensors, or scheduled watering.
- Migrating existing tray plants into locations automatically.
- Filtering bulk actions by species or status (whole-location only in v1).

## Data model

### New table `tray_location`

```sql
CREATE TABLE tray_location (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    org_id      BIGINT       NOT NULL REFERENCES organization(id) ON DELETE CASCADE,
    name        VARCHAR(255) NOT NULL,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_tray_location_org_id ON tray_location(org_id);
```

Just name. No description, no emoji, no kind enum — additive later if needed.

### New column on `plant`

```sql
ALTER TABLE plant
  ADD COLUMN tray_location_id BIGINT REFERENCES tray_location(id) ON DELETE SET NULL;
ALTER TABLE plant
  ADD CONSTRAINT plant_bed_or_tray_location_exclusive
  CHECK ((bed_id IS NULL) OR (tray_location_id IS NULL));
CREATE INDEX idx_plant_tray_location_id ON plant(tray_location_id);
```

A plant is in exactly one of three states: in a bed, in a tray at a known location, or in a tray with no location (legacy/orphaned). The check constraint forbids the impossible "in a bed AND at a tray location" state.

### New columns on `plant_event`

```sql
ALTER TABLE plant_event
  ADD COLUMN from_tray_location_id BIGINT REFERENCES tray_location(id) ON DELETE SET NULL,
  ADD COLUMN to_tray_location_id   BIGINT REFERENCES tray_location(id) ON DELETE SET NULL;
```

These are populated only for `MOVED` events. Nullable on either side: a NULL `from` means "previously had no location"; a NULL `to` means "moved out of all locations".

### New `PlantEventType` values

- `MOVED` — a plant changed tray locations. Sw label: "Flyttade" plural / "Flyttad" singular.
- `WATERED` — a plant was watered. Sw label: "Vattnade" plural / "Vattnad" singular.

## Picker behavior

Triggered from `SowActivityScreen` and `RegisterPlantsScreen` on Android, plus the equivalent on web later. Bed-rooted sowings never see the picker.

| Locations defined | Behavior                                                  |
| ----------------- | --------------------------------------------------------- |
| 0                 | No picker. `tray_location_id = NULL` on the plant.        |
| 1                 | No picker. Auto-assign the only location.                 |
| 2+                | Required picker with inline "+ Ny plats…" creation.       |

The inline create action opens a tiny dialog asking for a name only; on confirm, the location is created and selected.

## UI surfaces

### Drawer entry "Platser"

Under § Odling, after Trädgårdar, before Växter. Lists existing locations with name + active-plant count; create / rename / delete actions.

**Delete confirm** is count-aware: when the location has any active (`status != REMOVED`) plants, the prompt reads:

> 47 plantor i Greenhouse blir utan plats. Fortsätt?

Confirming detaches them (`tray_location_id = NULL`) and emits one `MOVED` event per plant with `to_tray_location_id = NULL`.

### Dashboard tray section

Grouped by location. Each group:
- Header: location name + total active count + inline quick-buttons (Vattna, Gödsla).
- Body: top species × status rows, like today's flat list, scoped to the location.

Plants without a location form a "Utan plats" pseudo-group with the same shape minus the bulk-action buttons (no location to scope them to).

Single-location accounts collapse to a single group; the layout still works.

### Location detail page

Reachable from the drawer entry, the dashboard group header, and the tray summary tap. Lightweight layout:

- Header: name + total count.
- Bulk-action buttons: **Vattna alla** · **Gödsla alla** · **Anteckna** · **Flytta**.
- Flat list of (species × status × count) rows, each tappable for a partial move.

The events expansion lives on the species detail page, not here. This page is a place lens, not a history lens.

## Actions

All bulk actions are whole-location scope (every plant currently `tray_location_id = X AND bed_id IS NULL AND status != REMOVED`).

### Vattna (water)

Persists one `WATERED` `plant_event` per plant with `event_date = today` and `plant_count = 1`. No supply consumption.

Backend endpoint: `POST /api/tray-locations/{id}/water`. Body optional (notes).

### Gödsla (fertilize)

Reuses the existing `ApplySupplyScreen` flow. Extension: the screen accepts a `tray_location_id` instead of (or in addition to) `bed_id`. The supply-deduction logic is unchanged; one `APPLIED_SUPPLY` event is recorded per plant in the location.

Backend: extend the existing apply-supply endpoint to take `trayLocationId`. The bed-or-tray-location is mutually exclusive in the request payload as well.

### Anteckna (note)

Persists one `NOTE` `plant_event` per plant with the supplied body.

Backend endpoint: `POST /api/tray-locations/{id}/note`. Body: `{ text: String }`.

### Flytta (move)

- **Whole-location**: top-level "Flytta alla" button on the location detail page. Picks a target location (or "(no location)") and moves every active plant.
- **Per-row partial**: tapping a (species × status) row offers a count-and-target picker, mirroring the existing "Ta bort N" dialog on the species detail.

Each move emits one `MOVED` `plant_event` per plant with `from_tray_location_id` / `to_tray_location_id` populated, and updates `plant.tray_location_id` to the new value.

Backend endpoint: `POST /api/tray-locations/{id}/move`. Body: `{ targetLocationId: Long?, count: Int, speciesId?: Long, status?: String }`. The optional `speciesId` and `status` filter the slice; absent = whole location.

## Render in events list

`TrayEventsExpansion` (the per-status events list inside the species detail page) extends to render `MOVED` events with the from/to context inline:

- `5 (10) st · Flyttade · från Greenhouse → Cold frame · 2026-04-26`
- `3 st · Flyttade · från Greenhouse → (ingen plats) · 2026-04-26`

`from`/`to` resolve via the FK at render time, so renaming a location updates historical events. This is the right behavior — the user cares about *that place*, not the historical name string.

`WATERED` and `NOTE` events render with their plural label and date, like other events. No location detail in the row itself; events are scoped to plants currently in a status, and a moved-out plant carries its history with it.

## Phasing

Three slices, each shippable on its own.

1. **Foundation** — schema + entity CRUD + drawer entry. No picker, no bulk actions yet.
2. **Sow integration** — picker on Sow + Register Plants, inline create, dashboard regrouping by location.
3. **Bulk actions** — location detail page + WATERED + bulk fertilize/note/move + delete confirm + render of MOVED in events expansion.

## Platforms

- **Android**: full feature, all three slices.
- **Web**: parity follows after Android slice 3 lands.
- **Admin**: not touched. Admin is for org/system management; tray locations are an operational concept.

## Open questions

None blocking. Future enhancements that came up but are out of scope:

- Location-typed defaults (`OUTDOOR` should not surface watering reminders during rain) — needs a kind enum and weather integration.
- Per-species bulk filtering ("water just the seedlings") — the v1 model is whole-location.
- Tray-as-entity (sub-divide a location into multiple physical trays) — significant model rework.
