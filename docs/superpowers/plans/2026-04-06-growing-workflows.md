# Growing Workflows Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add configurable growing workflows that define the step-by-step process for growing a species, from sowing to harvest. Plants track their progress through the workflow, and users can bulk-perform workflow steps.

**Architecture:** Workflow templates are standalone org-owned objects with ordered steps. Species get a copy of the steps when assigned a template. Plants track progress via a join table. Side-branches run alongside the main flow without blocking it.

**Tech Stack:** Quarkus/Kotlin backend, PostgreSQL with Flyway, React/TypeScript with React Query, i18n.

- [ ] **Step 1: Write the migration**

```sql
-- Workflow templates
CREATE TABLE workflow_template (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    org_id      BIGINT NOT NULL REFERENCES organization(id) ON DELETE CASCADE,
    name        VARCHAR(255) NOT NULL,
    description TEXT,
    created_at  TIMESTAMPTZ DEFAULT now(),
    updated_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_workflow_template_org ON workflow_template(org_id);

-- Workflow template steps
CREATE TABLE workflow_template_step (
    id                   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    template_id          BIGINT NOT NULL REFERENCES workflow_template(id) ON DELETE CASCADE,
    name                 VARCHAR(255) NOT NULL,
    description          TEXT,
    event_type           VARCHAR(50),
    days_after_previous  INTEGER,
    is_optional          BOOLEAN NOT NULL DEFAULT false,
    is_side_branch       BOOLEAN NOT NULL DEFAULT false,
    side_branch_name     VARCHAR(255),
    sort_order           INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_workflow_template_step_template ON workflow_template_step(template_id);

-- Species workflow steps (copied from template, editable per species)
CREATE TABLE species_workflow_step (
    id                   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    species_id           BIGINT NOT NULL REFERENCES species(id) ON DELETE CASCADE,
    template_step_id     BIGINT REFERENCES workflow_template_step(id) ON DELETE SET NULL,
    name                 VARCHAR(255) NOT NULL,
    description          TEXT,
    event_type           VARCHAR(50),
    days_after_previous  INTEGER,
    is_optional          BOOLEAN NOT NULL DEFAULT false,
    is_side_branch       BOOLEAN NOT NULL DEFAULT false,
    side_branch_name     VARCHAR(255),
    sort_order           INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_species_workflow_step_species ON species_workflow_step(species_id);

-- Link species to its source template (for re-sync)
ALTER TABLE species ADD COLUMN workflow_template_id BIGINT REFERENCES workflow_template(id) ON DELETE SET NULL;

-- Plant workflow progress
CREATE TABLE plant_workflow_progress (
    plant_id      BIGINT NOT NULL REFERENCES plant(id) ON DELETE CASCADE,
    step_id       BIGINT NOT NULL REFERENCES species_workflow_step(id) ON DELETE CASCADE,
    completed_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (plant_id, step_id)
);

CREATE INDEX idx_plant_workflow_progress_plant ON plant_workflow_progress(plant_id);
CREATE INDEX idx_plant_workflow_progress_step ON plant_workflow_progress(step_id);
```

- `assignTemplateToSpecies(speciesId, templateId, orgId)` — copies template steps to species
- `getSpeciesWorkflow(speciesId, orgId): SpeciesWorkflowResponse`
- `addSpeciesStep(speciesId, request, orgId): SpeciesWorkflowStepResponse`
- `updateSpeciesStep(stepId, request, orgId): SpeciesWorkflowStepResponse`
- `deleteSpeciesStep(stepId, orgId)`
- `syncSpeciesWithTemplate(speciesId, orgId): SpeciesWorkflowResponse` — re-copies from template, returns diff

**Plant progress:**
- `getPlantProgress(plantId, orgId): PlantWorkflowProgressResponse`
- `completeStep(speciesStepId, request: CompleteWorkflowStepRequest, orgId)` — bulk complete a step for multiple plants
  - Validates all plants are at this step
  - If step has eventType, creates PlantEvent for each plant
  - Records progress in join table
- `getPlantsAtStep(speciesId, stepId, orgId): List<PlantResponse>` — for bulk action UI
- `startSideBranch(plantIds, sideBranchName, orgId)` — marks plants as in a side-branch

**Side-branch logic:**
- A plant's active side-branches = side-branch names where at least one step is completed but not all steps in that branch are completed
- Starting a side-branch is just completing the first step of that branch
